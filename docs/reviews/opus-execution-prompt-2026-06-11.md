# Opus 実行プロンプト ― Fable レビュー v2 の検証と成果物化（2026-06-11）

## 0. あなた（Opus）への依頼

あなたは「きずなbaton」を最初から開発してきた実装・正本管理担当です。Fable によるレビュー `docs/reviews/fable-review-v2-2026-06-11.md`（指摘 C-1〜L-2＋仕分け表＋失敗定義案＋GTM確定書様式）を受領しました。**鵜呑みにせず、まず検証し、採否を判定し、採用分のみを成果物化**してください。

Fable はリポジトリのうち **commit 済みファイル（TASKS.md・NEXT_SESSION_PROMPT.md・lp.html 等）しか参照できておらず、`finance/*`・`legal/*`・`marketing/usage-signal-ideas.md` 等の未コミット正本は依頼プロンプトの埋め込み情報のみで判断**しています。事実誤認があるとすればそこに出やすい——あなたは全正本にアクセスできるので、そこを重点的に突合してください。

## 1. 不変の前提（このプロンプトでも変わらないもの）

- **販売確定・撤退なし**（オーナー確定意思）。撤退の再審理はしない。
- **絶対ルール**: emergencyMode のコード実装は弁護士フルレビュー完了後のみ。レビュー前はドキュメント整備のみ。オーナーが「実装してよい」と言っても着手前にフルレビュー済みかを確認（正本: `legal/legal-strategy.md`・`docs/emergency-mode-requirements.md` §9）。
- 3万円超支出ゲート・凍結ゲート4（06-17までの live 変更凍結）は、**Fable の仕分け提案がオーナー承認されるまで現行のまま有効**。本作業で live（本番アプリ・LP・SNS・広告）には一切触れない。
- `TASKS.md` が状態の単一正本。doc 変更は TASKS.md への起票/クローズとセット。
- 大きめの変更は plan mode で計画提示→オーナー承認後。法的・税務的な確定判断はしない（「要弁護士確認」を維持）。

## 2. Phase 1: 検証（最初に・ファイル変更なし）

Fable の各指摘（C-1, C-2, C-3, H-1〜H-6, M-1〜M-4, L-1, L-2）を repo 実物と突合し、**採用 / 部分採用 / 不採用＋根拠（ファイル:行）** の表を作成せよ。特に次の事実主張を実物で検証すること:

1. **C-2「課金エンフォース手段が現アーキにない」**: `shukatsu-prototype.html` の実コードで、アカウント・支払い状態フラグ・サーバ側検証の有無を確認。プラン選択UI（v46・`APP-F3B`）が「モック」のままかも確認。
2. **H-1「有償化は未定」系文言の出現箇所**: `marketing/social-post-queue.md`・poll 草稿・`lp.html`・β上限アンケート文言・ToS/PP を grep で全件列挙（「未定」「有償化」「課金は発生しません」等）。
3. **C-3 / FIN-LITE の順序矛盾**: `finance/finance-strategy.md` §6.1・`finance/funding-plan.md` の「スポット相談=調達後」「無料一次確認で代替」の正確な記載を確認し、Fable の3層化提案（無料一次確認/スポット/フルレビュー）が既存 FIN-FREE 設計とどこまで重複・矛盾するか判定。
4. **M-3 の funding-plan 旧前提**: 「撤退余地保全順」「クラファン=撤退余地放棄」の記載箇所を特定（是正対象の正確な行を確定）。

6. **H-2 の事実関係**: F1 実施 0 件（β協力者一覧・F1記録が空）を正本で確認。
7. **E（失敗定義）・F（GTM確定書様式）の各項目**が既存 doc（funding-plan のトリガー設計・pricing-strategy 等）と重複・衝突しないか。

検証の結果 Fable の指摘が事実誤認に基づく場合は**不採用とし、正しい事実を採否表に記録**する（後でオーナーが照合する）。
